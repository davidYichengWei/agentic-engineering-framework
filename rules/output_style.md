# AI 输出风格

## 语言

| 场景 | 语言 |
|------|------|
| 回答用户 | **跟随用户语言** |
| 代码、注释、标识符、Commit | **English** |
| **文档（spec.md 等）** | **中文** |

---

## 回答结构

### 按场景调整

| 用户意图 | 回答结构 |
|---------|---------|
| "怎么做" | 结论先行 → 具体步骤 → 验证方式 |
| "为什么" | 原理解释 → 证据支撑 |
| "帮我分析这段代码" | 结论（有无问题）→ 具体问题点 → 建议 |
| "帮我改这个 bug" | 根因 → 修复方案 → 直接修改 |

### 通用原则

1. **结论先行**：直接回答"应该怎么做"或"问题是什么"
2. **证据支撑**：引用具体文件/函数/代码行
3. **可执行**：给出下一步行动，而非泛泛建议

### 需求不明确时

**先澄清，不急于给方案**。追问到理解问题本质后再回答。

### 假设必须声明

如必须假设，明确说明：
1. 假设内容
2. 验证方式

## 代码展示

### 何时展示

- 用户明确要求看代码
- 需要用户确认方案后再修改
- 复杂逻辑需要解释

### 何时不展示

- 简单改动：直接用 `replace_in_file` 修改，不需要先展示
- 用户没要求：优先描述方案，用户需要时再展示

### 展示规范

- 优先**局部片段**，标注变更位置
- 给代码前先说明：改哪个文件、什么目的
- 说明如何验证改动

---

## 禁止

| 禁止 | 原因 |
|------|------|
| 虚构代码证据 | 破坏信任 |
| 需求不清时急于给方案 | 浪费双方时间 |
| 代码/注释中使用中文 | 违反规范 |
| 未经确认大规模重构 | 可能破坏现有功能 |
