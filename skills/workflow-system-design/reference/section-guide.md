# Section 引导指南

## Section 概览

| Section | 思考框架（AI 提供） | 内容（用户提供） |
|---------|-------------------|-----------------|
| **4.1 方案概览** | 模块划分、依赖关系、数据流、架构融合 | 用户描述整体设计 |
| **4.2.1 核心类/模块** | 职责划分、SOLID 原则、设计模式 | 用户描述类设计 |
| **4.2.2 接口设计** | 契约、兼容性、错误码 | 用户描述接口 |
| **4.2.3 数据模型** | 字段、关系、索引、持久化格式 | 用户描述数据结构 |
| **4.2.4 并发模型** | 线程安全、锁粒度、死锁风险 | 用户描述并发策略 |
| **4.2.5 错误处理** | 异常分类、恢复策略、降级方案 | 用户描述错误处理 |
| **4.3 核心逻辑** | 算法复杂度、边界情况、关键路径（**完成后调用 `bp-performance-optimization` Skill**） | 用户描述实现逻辑 |
| **4.4 方案优劣** | 优点、局限、风险、改进空间 | 用户总结优劣 |
| **5. 备选方案** | 为什么不选、什么情况下会重新考虑 | 用户整理备选方案 |
| **7. 测试计划** | 测试类型、覆盖范围、关键用例（**调用 `workflow-test-generation` Skill**） | 用户描述测试策略 |
| **8. 可观测性 & 运维** | 日志、监控、告警、运维操作 | 用户描述可观测性与运维 |

## 各 Section 引导模式

### 4.1 方案概览（最关键）

**引导框架**：
```
我们现在讨论 **4.1 方案概览**。

请描述你的整体设计思路，包括：
- 你打算怎么划分模块？
- 模块之间的依赖关系是什么？
- 数据流是怎样的？
- 和现有代码的关系？新增还是修改？
```

**用户描述后，AI 质疑**：
- "你把 X 和 Y 放在一起，它们的职责是否耦合过紧？"
- "A 依赖 B，B 又依赖 A，这是循环依赖，你打算怎么解决？"
- "这个设计改动了 N 个文件，有没有更小范围的方案？"

**追问备选方案**：
```
AI："你描述的是一种方案。你有考虑过其他方案吗？为什么选择现在这个？"
```

### 4.2 组件设计

按子节逐个讨论。每个子节：

```
AI："我们现在讨论 **4.2.1 核心类/模块设计**。

请描述你打算设计哪些类，它们的职责分别是什么？"
```

**用户描述后，AI 质疑**：
- "这个类做了 5 件事，是否违反单一职责？"
- "这两个类有重复逻辑，是否需要抽象公共基类？"
- "构造函数参数太多，是否考虑 Builder 模式？"

### 4.3 核心逻辑 + 性能优化（调用 Skill）

**引导框架**：
```
AI："我们现在讨论 **4.3 核心逻辑**。

请描述核心算法和实现逻辑：
- 关键路径是什么？
- 算法复杂度如何？
- 有哪些边界情况需要处理？"
```

**用户描述后，加载性能优化 Skill**：
```
AI："核心逻辑设计完成。我来加载 `bp-performance-optimization` Skill，从性能角度审视一下设计。"

[加载 bp-performance-optimization Skill]

AI："基于性能优化规范，我注意到几个可以优化的点：
- [具体优化建议]

你觉得这些优化有必要吗？"
```

### 6. 业界调研（AI 辅助）

这是唯一一个 AI 可以主动提供信息的 section：
```
AI："第 6 节业界调研，我来搜索一下相关的业界方案。"

[搜索后]

AI："我找到了几个相关的业界方案：
- X 系统用了 [方法 A]
- Y 系统用了 [方法 B]

你的设计和这些有什么异同？有什么可以借鉴的吗？"
```

### 7. 测试计划（调用 Skill）

**引导框架**：
```
AI："我们现在讨论 **7. 测试计划**。

我会加载 `workflow-test-generation` Skill 来帮助设计更完备的测试计划。"

[加载 workflow-test-generation Skill]

AI："基于测试生成规范，请描述你的测试策略：
- 需要哪些类型的测试？（单元测试、集成测试、MTR 测试）
- 关键的测试用例有哪些？
- 边界条件和异常场景？"
```

## 判断 Section 是否适用

对于 4.2.2-4.2.5、7、8 等 section：
```
AI："4.2.3 数据模型这个 section，你的需求涉及新的数据结构或持久化格式变更吗？如果不涉及可以跳过。"
```

**用户选择跳过时**：在 spec.md 该 section 标注 `N/A - 本需求不适用`。
