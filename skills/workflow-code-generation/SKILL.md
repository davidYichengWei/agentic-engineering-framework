# 代码生成

> **所有代码修改的统一入口。** 核心原则：**先加载规范，再写代码。**

## 触发条件

- 用户请求修改**代码文件**（新功能、优化、Bug 修复、重构）
- 用户说"帮我改/优化/重构这个..."且目标是代码文件
- 涉及代码文件变更的任何请求

**不触发场景**（无需调用本 skill）：
- 仅修改 `.md` 等文档文件
- 仅修改规则/配置类文本（且不涉及代码文件）

## 工作流程

### 步骤 1：评估复杂度，检查 spec.md

**先评估修改的复杂度**：

| 复杂度 | 信号 | spec 要求 |
|--------|------|----------|
| **简单** | 单文件、局部修改、明确的小 bug | 可跳过 spec 和 tasks.md |
| **中等** | 多文件、单模块、有一定影响范围 | 需要 spec + tasks.md |
| **复杂** | 跨模块、新特性、架构变更、大规模重构 | 需要完整 spec + tasks.md |

**对于中等及以上复杂度**：

查找：`docs/design-docs/<module>/<feature>/spec.md`

- **存在且完整** → 读取并确认理解
- **不存在或不完整** → **必须调用 `workflow-requirements-clarification` Skill**

> **禁止在 workflow-code-generation 内部自行澄清中等/复杂需求。**

**强制**：若 `spec.md` 与 `tasks.md` 同时存在，在编写任何代码前必须完整读取两者并纳入上下文。

**对于简单修改**：

可以跳过 spec，但必须先与用户确认修改目标和验证方式。

### Fast-Path：简单修改

**适用**：单文件、局部、小改动；或仅注释/空白/格式调整。**禁止**中等/复杂改动走此路径。

**步骤**：
1. 确认修改目标与验证方式
2. **加载编码规范**（同「步骤 3」，至少 `bp-coding-best-practices` + `bp-performance-optimization`）
3. 直接实现
4. 解释改动内容与原因

> 编译/验证由用户自行执行，除非用户明确要求。

### 步骤 2：检查/创建 tasks.md（核心步骤）

**对于中等及以上复杂度，必须有 `tasks.md`。**

#### 2a. 检查 tasks.md 是否已存在

- **已存在** → 读取 `tasks.md`，找到第一个未完成的 Task/子任务，进入「步骤 3」加载相关编码规范，再进入「步骤 4」开始实现
- **不存在** → 进入 2b 创建

#### 2b. 创建 tasks.md

在 spec.md 同目录下创建 `tasks.md`，模板见 [reference/tasks_template.md](reference/tasks_template.md)

**任务拆解原则**：

| 原则 | 说明 |
|------|------|
| **粒度** | 每个任务 < 200 行代码改动 |
| **独立性** | 每个任务可独立编译/测试 |
| **顺序** | 任务按依赖关系排序（被依赖的先做） |
| **可验证** | 每个任务有明确的完成标准 |

仅当新建 tasks.md 时，**必须向用户展示任务列表并确认**。

### 步骤 3：加载编码规范（强制前置步骤）

> ⚠️ **写任何代码前必须先加载规范。未加载就写代码 → 立即停止，先加载。**

**必须加载**：`bp-coding-best-practices` + `bp-performance-optimization`

**按需加载**：

| 规范 | 何时加载 |
|------|----------|
| `std-cpp` Skill | 文件为 `.cc`/`.cpp`/`.h` |
| `std-go` Skill | 文件为 `.go` |
| `bp-distributed-systems` Skill | 涉及网络通信、多节点协调、数据一致性、故障恢复 |

确认所有规范已加载后，进入步骤 4。

### 步骤 4：逐个任务实现

**⚠️ 核心规则：完成一个 Task → 输出报告 → 停止 → 等用户批准 → 才能继续下一个 Task**

#### 单个 Task 执行流程（必须严格遵循）

**Phase 1: 实现**
1. 声明：正在执行 Task N
2. 更新 tasks.md 标记当前任务为 In Progress
3. 修改代码

**Phase 2: 汇报**（完成实现后必须执行）
4. 输出 Task 完成报告（见下方模板）
5. 更新 tasks.md 标记 Completed

**Phase 3: 停止等待**（必须执行，不可跳过）
6. 停止执行，等待用户回复
7. 用户批准后，才能开始下一个 Task

#### Task 完成报告模板（必须使用）

**完成后，必须输出以下格式的报告**：

```markdown
---
## ✅ Task N 完成

### 改动文件
- `path/to/file1.cc`: [改动说明]
- `path/to/file2.h`: [改动说明]

### 改动内容
[简要说明做了什么，为什么这样做]

---
⏸️ **等待 Review**

请 review 以上改动。回复以下任一指令：
- **继续** / **下一个** / **LGTM** → 我将执行 Task N+1
- **修改** + 具体要求 → 我将修改后重新提交
- **回滚** → 我将撤销本次改动
```

#### 禁止行为

- 未加载编码规范就写代码
- 编译通过后直接开始下一个 Task（必须等用户批准）
- 不输出 Task 完成报告
- 一次完成多个 Task

#### 用户批准的判断

**以下回复视为批准**：
- "继续"、"下一个"、"LGTM"、"OK"、"好"、"可以"
- 任何表示同意的回复

**以下回复视为需要修改**：
- "修改 XXX"、"改一下 XXX"
- 任何包含具体修改要求的回复

**以下回复视为回滚**：
- "回滚"、"撤销"、"不要这个改动"

### 步骤 5：最终验证

所有任务完成后：
- 更新 tasks.md，标记所有任务状态
- 运行完整测试（如有）
- 总结整体改动

**清理 tasks.md**：

使用 `ask_followup_question` 询问用户：

```
问题：所有任务已完成，是否删除 tasks.md？
选项：
- [ ] 删除 → 删除 tasks.md
- [ ] 保留 → 保留 tasks.md 作为记录
```

## 用户跳过 spec 时

即使用户坚持跳过，仍必须生成简化版 spec.md（标注 `Status: Quick Draft`）。禁止在没有任何 spec 的情况下修改中等/复杂代码。

---

## 恢复中断的任务

如果之前的代码生成被中断：
1. 读取 `tasks.md`
2. 找到第一个未完成的 Task/子任务
3. **重新加载编码规范**（中断后需重新加载）
4. 从该 Task/子任务继续执行
