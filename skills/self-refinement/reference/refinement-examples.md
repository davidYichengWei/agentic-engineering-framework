# 经验沉淀场景示例

## 场景 1：规范缺失——命名约定未覆盖

### 错误过程

**用户请求**：为用户管理模块添加一个新的错误码常量

**AI 输出**：
```go
const ErrUserNotFound = 10001
const ErrUserAlreadyExists = 10002
```

**用户纠正**：
> 我们项目里错误码常量统一用 `ErrCode` 前缀 + 模块名，不是直接描述。应该是 `ErrCodeUserNotFound`。

### 诊断

- **根因类别**：规范缺失
- **分析**：现有的 `std-go` Skill 只规定了驼峰命名，但没有项目特定的错误码命名约定

### 沉淀建议

```markdown
### 建议 1: 补充错误码命名规范

- **根因**：规范缺失
- **目标文件**：`skills/std-go/SKILL.md`
- **操作**：在"核心规则速查"表格后新增一行
- **具体内容**：
  ```
  | **错误码** | 常量使用 `ErrCode` 前缀 + 模块名 + 描述，如 `ErrCodeUserNotFound` |
  ```
```

## 场景 2：流程遗漏——代码生成前未检查已有实现

### 错误过程

**用户请求**：实现一个 LRU 缓存

**AI 输出**：从零实现了一个完整的 LRU 缓存类

**用户纠正**：
> 项目的 `pkg/cache/` 下已经有一个通用的 LRU 实现了，你应该复用它而不是重写。

### 诊断

- **根因类别**：流程遗漏
- **分析**：`workflow-code-generation` Skill 中没有"实现前先搜索项目中是否有类似功能"的检查步骤

### 沉淀建议

```markdown
### 建议 1: 在代码生成流程中增加"复用检查"步骤

- **根因**：流程遗漏
- **目标文件**：`skills/workflow-code-generation/SKILL.md`
- **操作**：在"步骤 3"和"步骤 4"之间插入新步骤
- **具体内容**：
  ```
  ### 步骤 3.5：复用检查（强制前置步骤）
  
  在编写新代码前，搜索项目中是否已有类似实现：
  1. 搜索 `pkg/`、`lib/`、`common/`、`util/` 等公共目录
  2. 搜索与目标功能相关的关键词
  3. 找到类似实现 → 优先复用/扩展，不重写
  ```
```

## 场景 3：模式错误——用类比代替第一性原理

### 错误过程

**用户请求**：设计一个任务调度系统

**AI 输出**：
> 参考 Kubernetes 的 scheduler 设计，我们可以...

**用户纠正**：
> 不要直接搬 K8s 的方案。先搞清楚我们的约束：单机、最多 100 个任务、不需要分布式。这种量级用 K8s 的设计过度了。

### 诊断

- **根因类别**：模式错误
- **分析**：AI 直接用"业界方案类比"而非从问题本质出发。虽然 `thinking_principles.md` 已有第一性原理要求，但缺少"设计阶段"的具体检查点

### 沉淀建议

```markdown
### 建议 1: 在系统设计流程中强化约束确认

- **根因**：模式错误
- **目标文件**：`rules/thinking_principles.md`
- **操作**：在"协作实践"表格中追加一行
- **具体内容**：
  ```
  | "参考 X 系统设计" | 直接套用 X 的架构 | "X 处理的规模是多少？我们的规模和约束是什么？是否需要同等复杂度？" |
  ```
```
